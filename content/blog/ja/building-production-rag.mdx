ほとんどのRAGチュートリアルは "Hello World" で止まっている — PDFを読み込み、分割し、検索する。だが、本番のシステムはチュートリアルが触れないエッジケースで壊れる。複数文書にまたがるクエリ、意味を壊すチャンキング、もっともらしいが誤ったコンテキストを返す検索。

有料顧客向けにRAGを構築するとき、本当に重要なのは何か。実際のところを書く。

## 生成より検索が難しい

多くのRAGシステムで最も弱い部分はLLMではなく検索だ。間違ったチャンクを取ってきたら、どれだけプロンプトを工夫しても救えない。

```ts
// 悪い例: 固定長でナイーブに分割
const chunks = text.match(/.{1,1000}/g);

// 良い例: 見出し・リスト・コードブロックを
// 原子単位として扱う構造認識型の分割。
const chunks = splitBySemanticBoundaries(text, {
  maxTokens: 800,
  overlap: 120,
  respectBlocks: ["heading", "list", "code"],
});
```

ハイブリッド検索（BM25＋ベクトル）は、本番では純粋なベクトル検索よりほぼ常に優れている。キーワードのシグナルは思っている以上に効く。

## 評価は必須

感覚でRAGをチューニングすることはできない。評価セットを早めに作る — 50件のクエリと期待される回答があれば十分に始められる。検索のリコールをエンドツーエンドの回答品質と分けて計測する。検索が下がればシステム全体が下がる。

## 人をループに入れる

高リスクな回答には、信頼度の低い応答を人間のレビュアーに回すルートを用意する。信頼度は検索スコア、LLMの自己評価、質問タイプのヒューリスティックから算出できる。

RAGを成功させるチームは、これをプロンプトエンジニアリングではなくシステム設計の問題として扱っている。
